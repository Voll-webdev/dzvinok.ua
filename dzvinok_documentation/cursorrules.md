# Правила розробки проекту dzvinok.ua

## Загальні правила

### Робота з базою даних
- Всі міграції створюються виключно на стороні сайту в каталозі dzvinok_main
- Всі таблиці в базі даних іменуються в PascalCase (Products, Categories, ProductCategories, Orders, OrderItems, Users тощо)
- Адмін панель (dzvinok_admin) використовує моделі та ресурси Filament для роботи з БД, але не створює міграції

### Структура основного сайту (dzvinok_main)
- Основний сайт має модульну структуру бекенду
- Всі основні розділи виділяються в окремі модулі в каталозі `dzvinok_main/src/`
- Кожен модуль містить власні контролери, сервіси, запити (requests), шаблони (views) та інші компоненти
- Всі моделі Eloquent створюються централізовано в `dzvinok_main/app/Models/` (не в модулях)
- Приклад структури модуля:
  - `src/Catalog/` - модуль каталогу товарів
  - `src/Cart/` - модуль кошика
  - `src/Order/` - модуль замовлень
  - `src/Auth/` - модуль авторизації
  - `src/Profile/` - модуль особистого кабінету

### Структура документації
- Під кожен розділ створюється окрема папка в dzvinok_documentation:
  - `admin/` - документація адміністративної панелі (products.md, categories.md, orders.md, filters.md тощо)
  - `main/` - документація основного сайту
  - `api/` - документація API для інтеграції з 1С
  - `TODO/` - спринти та загальний TODO файл
- Кожен модуль документується в окремому файлі або у файлі відповідного розділу
- Вся документація повинна бути строгою, без зайвої інформації
- Заборонено використання емодзі в документації

### Процес розробки
- Імплементація функціоналу починається лише після затвердження всієї документації
- Після кожного етапу виконаних робіт обов'язково тестується весь реалізований функціонал
- Зміни в базі даних супроводжуються оновленням відповідної документації

### Контроль версій
- Всі зміни повинні бути описані в commit message
- Перед початком роботи над новим функціоналом створюється нова гілка
- Злиття в main гілку відбувається після тестування

### API
- Всі API endpoints для інтеграції з 1С знаходяться в dzvinok_main
- API забезпечує двосторонній обмін даними з 1С
- Кожен endpoint документується в каталозі `api/`

### Naming conventions
- Файли міграцій: `YYYY_MM_DD_HHMMSS_create_table_name_table.php`
- Моделі: PascalCase (Product, Category, Order)
- Контролери: PascalCase з суфіксом Controller (ProductController)
- Методи: camelCase (getProducts, createOrder)
- Змінні: camelCase ($productList, $orderTotal)
